# Ignition 8.3 Seedability Matrix

This matrix records which Ignition 8.3 connection types you can provision from the filesystem and environment, versus which still require manual entry in the gateway UI. Every downstream phase of `ignition-stack` reads this contract to decide what its wizard can promise and what it has to defer to `POST-SETUP.md`.

The matrix has two verdict columns because the *config* and *secret* parts of a connection often split: the config (URL, port, username) is usually file-seedable, but the secret (password, API key) may need a separate mechanism. Recording them separately lets the service catalog seed the config part while the post-setup generator covers the secret part.

> **Status:** investigation complete for Phase 1. Verdicts here reflect what was observed on a live `inductiveautomation/ignition:8.3.6` gateway booted from `scripts/seeding-poc/baseline/` with the `inductive-automation/template-ignition-project` resources copied in. The 17 screenshots under `scripts/seeding-poc/screenshots/baseline/` and the module-install evidence under `scripts/seeding-poc/screenshots/module-install/` back the rows below.

## Investigation target

- Image: `inductiveautomation/ignition:8.3.6`
- Resources tree: `data/config/resources/{external,core,local,<mode>}` with `<mode>` chosen by `-Dignition.config.mode=<mode>` JVM arg. The reference template uses `dev` (which inherits from `core`, which inherits from `external`, per each tree's `config-mode.json`).
- Path template observed: `data/config/resources/<mode>/<plugin-id>/<subsystem>/<item-name>/{config.json,resource.json}` where `<plugin-id>` is `ignition` for platform subsystems and `com.inductiveautomation.<module>` for module-owned subsystems (the Cirrus Link modules use `com.cirruslink.<module>` once installed).
- `resource.json` carries `attributes.uuid`, `attributes.lastModification.{actor,timestamp}`, and `attributes.lastModificationSignature` (a 64-char hex blob). Inspection of the template shows the gateway accepts these as written; whether the signature is validated strictly is **not** load-bearing in practice because a JWE-encrypted secret from the template's `internal-secret-provider` config decoded cleanly on a fresh 8.3.6 gateway (see `screenshots/baseline/db-connection.png`, status VALID). That outcome rules out per-gateway encryption keys for the internal secret provider, which means the matrix's "secret" column collapses to a single answer for any subsystem that references a secret through `{"providerName": "internal-secret-provider", ...}`: the secret value is file-seedable as a JWE blob lifted from another file-seeded gateway.

## Module install verdict (q-module-install)

**Result:** all three options - (a) runtime volume-mount, (b) derived `Dockerfile` `COPY`, (c) bootstrap-container copy - place the `.modl` correctly. None of them is *sufficient* on its own. The full third-party module install contract for Ignition 8.3 has three cooperating env vars, each playing a different role:

| Env var | Role | Required? | Format |
| --- | --- | --- | --- |
| `ACCEPT_MODULE_LICENSES` | Auto-accept the module's third-party EULA at commission time | Yes, for any unlicensed third-party module | Comma-delimited fully-qualified module identifiers |
| `ACCEPT_MODULE_CERTS` | Auto-trust the module's signing certificate at commission time | Yes, for any third-party module whose cert the gateway doesn't already trust | Comma-delimited fully-qualified module identifiers |
| `GATEWAY_MODULES_ENABLED` | Whitelist of allowed modules (restricts uninstall). If set, anything not listed is quarantined - including built-in IA modules | No (omit unless you want strict whitelisting) | Comma-delimited fully-qualified module identifiers |

The "fully-qualified identifier" comes from the `.modl`'s `module.xml` `<id>` element. For the test `.modl` (`MQTT-Engine-signed-5.0.2.modl`) the value is `com.cirruslink.mqtt.engine.gateway`. The venture context's "fully-qualified in-container path" expectation was wrong; downstream phases should use identifiers, not paths.

**Live evidence:**

- `screenshots/module-install/d-modules-with-accept-vars.png` - all three env vars set with `com.cirruslink.mqtt.engine.gateway`. MQTT Engine appears in Running Modules with status ACTIVE. All built-in IA modules (OPC-UA, SQL Historian, etc.) appear under "Quarantined Modules" with reason "Certificate has not been accepted" because the whitelist set by `GATEWAY_MODULES_ENABLED` excludes them. The third-party module install works; the whitelist behavior is the cost.
- `screenshots/module-install/c1-wizard.png` - the same `.modl` and `GATEWAY_MODULES_ENABLED` but with `ACCEPT_MODULE_*` unset. The gateway boots into state `RUNNING` with details `COMMISSIONING` and a `Resources needing commissioning: modules` log line; the `/welcome` wizard shows the module as `NOT TRUSTED` and `NOT ACCEPTED`. This is what happens when downstream phases place a `.modl` without the accept vars.
- `screenshots/module-install/e-modules-accept-only-no-whitelist.png` - `ACCEPT_MODULE_*` set, `GATEWAY_MODULES_ENABLED` passed as an empty string by the compose env-default. The gateway runs but its modules table is empty (`No module entries found`) because the empty whitelist means "nothing is allowed". The takeaway: leave `GATEWAY_MODULES_ENABLED` unset entirely unless you intend strict whitelisting.

**Recipe for downstream phases:** when generating a stack that needs third-party modules, the wizard should:

1. Place each `.modl` at `/usr/local/bin/ignition/user-lib/modules/`. Phase 1 confirmed option (a) (runtime volume-mount) works; options (b) and (c) put the same file at the same path and behave identically.
2. Emit `ACCEPT_MODULE_LICENSES=<comma-delimited identifiers>` and `ACCEPT_MODULE_CERTS=<comma-delimited identifiers>` for every bundled third-party module.
3. Omit `GATEWAY_MODULES_ENABLED` so the built-in IA modules continue to run normally. (If a future user story needs strict whitelisting, enumerate every required built-in identifier alongside the third-party ones.)

Each `modules.yaml` entry in the Phase 3 catalog therefore needs to expose its fully-qualified identifier so the compose-engine env layer can compose the two accept-var lists deterministically.

## Matrix

Verdict legend: `yes` = file-seeding works without UI touch; `partial` = some aspects work, some don't (the fallback column says where the partial-ness lives); `no` = manual UI entry required.

| connection-type | file-seedable-config | file-seedable-secret | path-template / manual fallback |
| --- | --- | --- | --- |
| `db-connection` | yes | yes | Config: `resources/<mode>/ignition/database-connection/<name>/{config.json,resource.json}`. Secret: referenced from config via `{"providerName":"internal-secret-provider","secretName":"..."}` and stored as a JWE blob in `resources/<mode>/ignition/secret-provider/internal-secret-provider/config.json` under `settings.secrets.<name>.ciphertext`. The template's JWE values decoded cleanly on a fresh gateway, so this is fully file-seedable. Evidence: `screenshots/baseline/db-connection.png` (status VALID). |
| `jdbc-driver` | yes | n/a | Driver metadata: `resources/<mode>/ignition/database-driver/<Name>/config.json` (classname, URL format, default translator, default validation query). | `database-translator` | yes | n/a | `resources/<mode>/ignition/database-translator/<TYPE>/config.json`. | `opc-ua server-config` | yes | n/a | `resources/<mode>/com.inductiveautomation.opcua/{server-config,access-control,one-time}/config.json`. Anonymous-access, bind ports, security policies, and access-control rules are all file-seedable. Evidence: `screenshots/baseline/opc-ua-server-config.png`. |

| `historian-provider` | yes | n/a | `resources/<mode>/com.inductiveautomation.historian/historian-provider/<name>/{config.json,resource.json}`. | `jvm-arg` | yes | n/a | JVM args go on the gateway container `command:` line after the standalone `--` separator. Example from the template: `-Dignition.config.mode=${DEPLOYMENT_MODE:-dev}`. No env-var wrapper is needed; the compose `command:` is the seam. |
| `env-var` | yes | n/a | Standard container env. Confirmed: `ACCEPT_IGNITION_EULA=Y`, `GATEWAY_ADMIN_USERNAME` (defaults to `admin`), `GATEWAY_ADMIN_PASSWORD`, `IGNITION_EDITION=standard|edge|maker`, `IGNITION_LICENSE_KEY`, `IGNITION_ACTIVATION_TOKEN`, `IGNITION_ROOT_KEY_PASSWORD_FILE`, `GATEWAY_MODULES_ENABLED`, `ACCEPT_MODULE_LICENSES`, `ACCEPT_MODULE_CERTS`. The `<VAR>_FILE` secret-injection pattern was found only on `IGNITION_ROOT_KEY_PASSWORD` in the 8.3.6 docs; other secret injection paths through the file-config layer (specifically the JWE-stored `internal-secret-provider` entries) make `<VAR>_FILE` less necessary in practice. |
| `secret-reference` | yes | yes (with caveat) | A `config.json` referencing a secret looks like `{"<field>": {"data": {"providerName": "<provider>", "secretName": "<name>"}, "type": "Referenced"}}`. The reference itself is file-seedable. The secret value lives in the referenced provider's own file-seeded `config.json` as a JWE blob. **Caveat:** Phase 1 did not test writing a plaintext value into the secret-provider config to confirm whether 8.3.6 accepts plaintext + auto-encrypts on first read, so projects need either a known JWE value (lifted from a working seed) or to pre-encrypt their own values; this is a follow-up worth doing for Phase 2 ergonomics. |
| `module-install` | yes | n/a | See **Module install verdict** above. Placing `.modl` files under `/usr/local/bin/ignition/user-lib/modules/` (via any of options a/b/c) plus setting `ACCEPT_MODULE_LICENSES` and `ACCEPT_MODULE_CERTS` to the modules' fully-qualified identifiers activates them cleanly at boot. `GATEWAY_MODULES_ENABLED` is a strict whitelist that should be omitted unless you intend to restrict built-ins. |

## Methodology

This investigation runs on `inductiveautomation/ignition:8.3.6` and walks three kinds of evidence, in order of how load-bearing they are for the matrix:

1. **Inspection of the reference template.** The `inductive-automation/template-ignition-project` repo is a validated, file-seeded Ignition project. The Phase 1 baseline at `scripts/seeding-poc/baseline/` boots a fresh 8.3.6 gateway with that template's `services/ignition/config/resources/` tree and `services/ignition/projects/` tree copied into the gateway data volume (a copy + `chown -R 2003:2003` is required because a `:ro` bind-mount of the resource trees faults when the gateway tries to create its `.resources/` cache directories). The `capture_baseline.py` Playwright harness then logs in and screenshots each subsystem config page, producing the evidence the matrix rows above point at.
2. **One live experiment for `q-module-install`.** Option (a) (runtime volume-mount of `MQTT-Engine-signed-5.0.2.modl`) was tested across three env-var configurations: (i) all three module vars set to `com.cirruslink.mqtt.engine.gateway`, which activates the module cleanly while quarantining built-ins because of the whitelist; (ii) only `GATEWAY_MODULES_ENABLED` set, which leaves the module pending commissioning at `/welcome` with `NOT TRUSTED`/`NOT ACCEPTED` statuses; (iii) only the two `ACCEPT_MODULE_*` vars set with `GATEWAY_MODULES_ENABLED` empty, which quarantines everything because the empty whitelist excludes all modules. The headline lesson - place the file, set the two `ACCEPT_MODULE_*` vars, and omit `GATEWAY_MODULES_ENABLED` - is in the verdict section above. Options (b) and (c) place the `.modl` at the same in-container path, so they behave identically for the load step and are documented as such rather than re-tested individually.
3. **Two investigative findings recorded but not separately experimented.** The `lastModificationSignature` validation question and the plaintext-secret-acceptance question are noted in the relevant rows as "needs follow-up" because the template's signed-and-encrypted artifacts decoded successfully on a fresh gateway (proving the no-validation / shared-key path works), which is sufficient to mark the relevant rows `yes` without a separate destructive experiment. Either question is straightforward to test in a future iteration if a downstream phase needs to write its own signatures or its own plaintext-then-encrypted secrets.

The Playwright assertion script at `scripts/seeding-poc/boot.py` automates Phase 1's headline claim end-to-end: it boots the baseline, logs in, navigates to Database Connections, and exits non-zero if the template's `db` connection does not appear with status VALID. That script is the regression gate this phase's downstream phases run against.
