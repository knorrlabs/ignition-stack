# Redundancy spike: how an 8.3 master/backup pair comes up in Docker

Hands-on findings from the Phase 3 redundancy spike (venture
`declarative-redundancy-ignition-stack-generation`). Everything here was
verified against two real `inductiveautomation/ignition:8.3.6` containers on a
shared Docker network, driving the gateway UI by hand. This is a research
artifact, not a CLI harness: nothing here is generated by or consumed by
`ignition-stack`. Its job is to decide, with evidence, what Phase 4 should do
(env-only wiring vs. env + a seeded `redundancy.xml`).

## TL;DR for Phase 4

**A redundant pair can be brought up fully zero-touch** — master and backup boot
already paired, with no UI clicks — by combining two file/env-seedable things:

1. **An open incoming Gateway Network policy** on both nodes
   (`securityPolicy=Unrestricted`, `requireSSL=false`), set via env vars or the
   `gateway-network-settings` resource file.
2. **A pre-seeded `data/redundancy.xml`** on each node carrying the redundancy
   *mode* (master vs. backup + the master's address), using a **plain (non-SSL)**
   redundancy connection on **port 8088**.

Plain transport is the load-bearing choice: it sidesteps the Gateway Network
**server-certificate trust handshake**, which is the *only* step that otherwise
forces a manual "Approve" click. This matches the `publicdemo-all` dev stack
(GAN on 8088, no SSL, `Unrestricted`).

**Recommendation: Phase 4 should seed `redundancy.xml` (plain/8088) plus the
open GAN policy, and treat full pairing as automatic.** The "env-only, no
`redundancy.xml`" path the design hoped for does **not** pair — it only builds
the Gateway Network transport; the redundancy *role* is never set. Keep the
manual runbook (below) as the fallback/explainer and for the SSL variant.

**Security scope:** plain / `requireSSL=false` / `Unrestricted` is acceptable
here *because the demo stack is a single isolated Docker network* — the gateway
itself logs `Require SSL setting is disabled, rendering Gateway Network
connections insecure` when you do this. Redundancy replicates the full system
backup (projects, tags, secrets) over that link, so any cross-host or
production deployment should use SSL (8060) + approved certificates instead.
Phase 4 should generate plain-by-default for the demo, but say this out loud in
the generated docs.

## What's in here

| File | Purpose |
| --- | --- |
| `docker-compose.yaml` | Minimal default-path stack: two vanilla 8.3.6 gateways, no Gateway Network env at all. Used to record the **manual pairing runbook** (you configure redundancy by hand and watch which steps the platform forces). |
| `docker-compose.seed.yaml` | **Zero-touch stack**: open GAN policy via env + per-node init containers that pre-seed `redundancy.xml`. Boots already paired. |
| `seed/master-redundancy.xml`, `seed/backup-redundancy.xml` | The seed files dropped into each data volume. Master = `noderole=Master`; backup = `noderole=Backup`, `gan.host=master`, `gan.port=8088`, `gan.enableSsl=false`. |
| `captured/` (gitignored) | Live state pulled off the running containers during the spike. The canonical observed contents are transcribed below. |

```sh
# zero-touch: comes up already paired (PEER CONNECTED: Yes within ~1 min of RUNNING)
docker compose -f docker-compose.seed.yaml up -d
# master UI http://localhost:9088  backup UI http://localhost:9089  (admin / password)

# manual runbook baseline (configure redundancy by hand per the runbook below)
docker compose up -d

docker compose [-f docker-compose.seed.yaml] down -v   # always -v to drop the volumes
```

## Ports

| Port | Role |
| --- | --- |
| `8088` | Gateway HTTP **and** plain (non-SSL) Gateway Network. Redundancy can ride this when `enableSsl=false`. |
| `8060` | SSL/TLS Gateway Network. The **default** redundancy connection port (`redundancy.gan.port`, `enableSsl=true`). |

The redundancy backup connects to the master *over the Gateway Network* using
its own `redundancy.gan.*` connection params (separate from any generic
`GATEWAY_NETWORK_#_*` outgoing connection). In the spike the host mapped
`9088->8088` (master) and `9089->8088` (backup) for the UIs; the GAN traffic
between containers stays internal on the shared network.

## Findings

### 1. There is still no redundancy-commissioning env var

Confirmed: the image exposes nothing to set redundancy *mode* via env. Booting
with only `GATEWAY_NETWORK_0_HOST=master` (the backup pointed at the master)
leaves the backup's role at **Independent**. The env vars build the Gateway
Network *transport*; they do not make a node a backup.

### 2. The Gateway Network incoming policy IS env- and file-seedable

The 8.3 image honors a `GATEWAY_NETWORK_*` policy family (not documented on the
Docker page; found on the Platform Environment Variables reference):

| Env var | Effect | Default |
| --- | --- | --- |
| `GATEWAY_NETWORK_ALLOWINCOMING` | accept inbound GAN connections | — |
| `GATEWAY_NETWORK_SECURITYPOLICY` | `ApprovedOnly` \| `Unrestricted` \| `SpecifiedList` | `ApprovedOnly` |
| `GATEWAY_NETWORK_REQUIRESSL` | require SSL for incoming | `true` |
| `GATEWAY_NETWORK_REQUIRETWOWAYAUTH` | require mutual TLS | — |
| `GATEWAY_NETWORK_#_HOST` / `_PORT` / `_ENABLESSL` | a generic outgoing connection (`_PORT` default `8060`, `_ENABLESSL` default `true`) | — |

These land in `data/config/resources/<mode>/ignition/gateway-network-settings/config.json`
(verified: `securityPolicy=Unrestricted`, `requireSSL=false`, `allowIncoming=true`),
so the policy is equally seedable as a resource file — which is exactly what the
`publicdemo-all` stack ships.

### 3. The 8.3 redundancy backing store is `data/redundancy.xml` (the 8.1 path holds)

No `redundancy.xml` exists until redundancy is configured (a fresh gateway only
has the stock `System/Gateway/Redundancy` *tag* definitions). The moment you
save a redundancy mode in the UI, **`data/redundancy.xml`** and a
`.redundancy.xml.bak` appear in the data root — **not** under `resources/<mode>/`.
It is a Java-properties XML. Observed schema (8.3.6), master vs. backup differ
only in `noderole` and `gan.host`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE properties SYSTEM "http://java.sun.com/dtd/properties.dtd">
<properties>
<comment>Redundancy Settings</comment>
<entry key="redundancy.activehistorylevel">Full</entry>
<entry key="redundancy.autodetectlocalinterface">true</entry>
<entry key="redundancy.bindinterface"></entry>
<entry key="redundancy.gan.enableSsl">true</entry>          <!-- false for plain/8088 -->
<entry key="redundancy.gan.host"></entry>                   <!-- backup: the master's address -->
<entry key="redundancy.gan.httpConnectTimeout">10000</entry>
<entry key="redundancy.gan.httpReadTimeout">60000</entry>
<entry key="redundancy.gan.pingMaxMissed">10</entry>
<entry key="redundancy.gan.pingRate">1000</entry>
<entry key="redundancy.gan.pingTimeout">300</entry>
<entry key="redundancy.gan.port">8060</entry>               <!-- 8088 for plain -->
<entry key="redundancy.gan.receiveThreads">5</entry>
<entry key="redundancy.gan.sendThreads">5</entry>
<entry key="redundancy.gan.websocketTimeout">10000</entry>
<entry key="redundancy.joinwaittime">30000</entry>
<entry key="redundancy.masterrecoverymode">Automatic</entry>
<entry key="redundancy.noderole">Independent</entry>        <!-- Master | Backup | Independent -->
<entry key="redundancy.standbyactivitylevel">Cold</entry>
<entry key="redundancy.sync.timeoutSecs">60</entry>
<entry key="redundancy.useActiveUptimeToResolveConflict">false</entry>
</properties>
```

**Seeding takes.** Dropping these files in before first boot puts each node
straight into its role; the gateway preserves the seeded file (re-read after
pairing: still `noderole=Backup`, `gan.port=8088`, `enableSsl=false`).

### 4. The default GAN policy rejects plain, and SSL needs a cert approval

A fresh master's policy is `requireSSL=true` + `securityPolicy=ApprovedOnly`. So:

- **Plain (8088) into a default master → `403 Forbidden`** (SSL is required).
- **SSL (8060, the redundancy default) → `certificate_unknown`.** The backup
  refuses the master's self-signed GAN server cert:
  > `No approved certificate could be found... the server certificate 'CN=master:8060, OU=GatewayNetwork...' must be added to this Gateway.`
  This is **backup-side** trust (the node initiating the outbound connection
  must trust the remote's server cert). `Unrestricted` + `requireSSL=false` does
  **not** auto-trust an SSL cert — cert trust is a separate PKI axis. Approving
  the cert is the manual step (control located below).

### 5. Env-only forms the transport but never the pair

With the backup's generic `GATEWAY_NETWORK_0_*` pointed at the master and an
`Unrestricted`/no-SSL policy, the Gateway Network connection auto-establishes
and auto-approves (`Ping to remote machine was successful`, ~10-12 s after both
nodes are RUNNING) — but both nodes stay **Independent**. No `redundancy.xml`,
no pair. This is the design's "env-only" hypothesis, and it is **insufficient**.

### 6. Zero-touch pairing works with open policy + seeded `redundancy.xml` (plain/8088)

`docker-compose.seed.yaml` boots master+backup that pair with **no UI at all**.
Within ~1 minute of both reaching RUNNING the master shows:

> **Gateway Role: Master · PEER CONNECTED: Yes · Activity Level: Active · Sync
> Status: Good · Peer Address: master-Backup**

and the backup pulls a full config sync from the master. Certs sync
automatically over the plain transport (`Syncing certificates from diffs
returned from peer`) — no approval. This is the recommended Phase 4 shape.

### 7. The backup must not be renamed (`-n`)

On pairing, the backup **adopts the master's system name** (observed peer name
`master-Backup`, derived from master `master`). The IA docker image warns
against `-n` on a backup for exactly this reason. The seed stack therefore omits
`-n` on the backup service and keeps it on the master.

> **Phase 4 note:** this collides with the standalone template's
> `scripts/docker-bootstrap.sh`, which derives the deterministic GAN UUID from
> `GATEWAY_NAME` and whose compose passes `-n ${GATEWAY_NAME}`. The backup node
> can't reuse that path unchanged — it needs its own GAN UUID but **no** `-n`
> rename (its system name is overwritten by the master's on first sync anyway).
> Either special-case the backup in the bootstrap/compose rendering or drop `-n`
> for the backup specifically.

### 8. Edge and licensing (from 8.3 docs; not stood up)

- **Edge redundancy is Edge-to-Edge only.** "An Edge Gateway can only failover
  to another Edge Gateway (not a standard Ignition Gateway)." A standard-master
  / Edge-backup pair is therefore invalid — Phase 4's resolver must reject it.
- **Both nodes need a license.** A discounted **backup-only** license **forces**
  the node into Backup mode (the Mode dropdown is disabled while applied).
- **Unlicensed = 2-hour trial** (confirmed live: the spike gateways ran in
  "Trial Mode" with a ~2 h countdown). Trial is enough to set modes and observe
  pairing; failover longevity needs licenses. The seedability findings hold
  regardless of licensing.

## Manual pairing runbook (default / SSL path)

Use this when you don't pre-seed (or to explain what the seed automates). Both
gateways must be on the same Gateway Network and reachable by hostname.

1. **Master** → `Config ▸ Platform ▸ System ▸ Redundancy` →
   **Configure Redundancy** → set **Mode = Master** → **Save Changes**.
   (Role flips to Master, Activity Level → Active.)
2. **Backup** → same page → **Configure Redundancy** → set **Mode = Backup**,
   **Master Node Address =** the master's hostname/service name. Leave **Port
   8060 / Use SSL on** for the default secure path (or **uncheck Use SSL / Port
   8088** for plain). → **Save Changes**.
3. **(SSL path only) Approve the master's certificate on the backup.** The
   backup's outbound SSL connection fails with `certificate_unknown` until you
   trust the master's GAN server cert:
   `Config ▸ Network ▸ Gateway Network ▸ Connections` → **Outgoing Connections**
   tab → **Remote Certificates** → the `master:8060` row shows **Not Approved**
   → row menu → **Approve**.
   (Plain/8088 needs no cert step, but the master must allow non-SSL incoming —
   i.e. `requireSSL=false`.)
4. Confirm on the master's Redundancy page: **PEER CONNECTED: Yes**, **Sync
   Status: Good**. The backup performs a one-time full config pull and runs as
   Cold (or Warm) standby.

> **Evidence note:** `PEER CONNECTED: Yes / Sync Status: Good` was confirmed
> end-to-end only on the **plain/zero-touch** path (`docker-compose.seed.yaml`).
> On the SSL path the spike confirmed that cert approval is the gate and located
> the **Approve** control, but did not observe the SSL connection complete to
> `PEER CONNECTED: Yes` after approval — the faulted redundancy connection retries
> on its ping cycle and may need the backup gateway (or the redundancy connection)
> restarted to pick up the newly trusted cert. Treat step 3 as "approve, then
> allow a reconnect / restart," not an instantly-verified one-click.

Timings (image already cached): first boot to RUNNING ≈ 2 min/node; manual
config ≈ 30-60 s of clicking per node; pairing completes within seconds of the
backup's mode being saved (or of both nodes booting, for the seeded stack).
