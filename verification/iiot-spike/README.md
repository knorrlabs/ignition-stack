# IIoT spike: is the Cirrus Link Sparkplug pipeline file-seedable?

Hands-on findings from the Phase 4 IIoT seeding spike (issue #43). Everything
here was verified against real `inductiveautomation/ignition:8.3.6` containers,
Cirrus Link Transmission/Engine modules `5.0.3`, and `cirruslink/chariot:3.0.0`
on a shared Docker network (2026-06-11). This is a research artifact: the
verdict feeds the writer, but nothing here is generated by or consumed by
`ignition-stack`. Its job was to decide, with evidence, whether Phase 4 seeds
the MQTT connection or defers it to `POST-SETUP.md`.

## TL;DR verdict

**PASS — the Cirrus Link MQTT Transmission/Engine connection config IS
file-seedable, zero-touch.** A volume-wiped hub-and-spoke stack booted with
captured seeds under
`config/resources/core/com.cirruslink.mqtt.transmission.gateway/…` (the spoke)
and `…mqtt.engine.gateway/…` (the hub) paired with no UI:

- the spoke logged `Connected to the MQTT Server`, then
  `Publishing NBIRTH on Topic: spBv1.0/iiotspike/NBIRTH/spoke-1` and the DBIRTH;
- a live broker capture showed `spBv1.0/STATE/IamHost {"online":true}` (the
  Engine's primary host) and `spBv1.0/iiotspike/DDATA/spoke-1/Gateway` carrying
  real tag values.

The seeded Sparkplug identity — **Group ID `iiotspike` = the project name, Edge
Node ID `spoke-1` = the gateway name** — came entirely from the seed files, which
is exactly the identity convention Phase 4 templates (`groupId = {{ project }}`,
`edgeNodeId = {{ gateway }}`, `url = tcp://{{ broker }}:{{ port }}`).

**The JWE-encrypted MQTT passwords in the seeds are portable across gateway
keystores** — captured on one gateway, decrypted on a fresh one — the same
property already proven for the `db-connection` and Keycloak OIDC seeds.

**Recommendation, now implemented:** the broker grows role-scoped
gateway-resource trees that the writer places by attachment role (Transmission
onto transmission-attached gateways, Engine onto engine-attached), with the
identity/URL templated per gateway. The shared trees live in
`ignition_stack/templates/iiot/` so every broker that wires the same Cirrus
modules reuses them.

| Path | Purpose |

## Findings

| # | Finding | Verdict |
| --- | --- | --- |
| 1 | Transmission gateway config (server connection, transmitter/edge-node, server-set, history-store, general) is fully file-seedable under `com.cirruslink.mqtt.transmission.gateway/`. | seedable |
| 2 | Engine gateway config (server connection, namespace, server-set, general) is fully file-seedable under `com.cirruslink.mqtt.engine.gateway/`. Engine needs no Sparkplug identity — it subscribes to `spBv1.0/#`. | seedable |
| 3 | The Sparkplug Group ID + Edge Node ID ride the transmitter item (`groupId`, `edgeNodeId`); the `transmitter/<edge-node>/` directory is named for the edge node. Both came from the seed, no UI. | seedable |
| 4 | The broker connection password is an **embedded JWE blob** (`{"type":"Embedded","data":{<JWE>}}`) inside the server `config.json`; it decrypted on a fresh gateway keystore. | portable |
| 5 | **Chariot license gate (dead end avoided):** `chariot:3.0.0` refuses to open its MQTT listener without a license — `Not starting Chariot MQTT Server, license not active`. No amount of gateway-side seeding helps until the broker is licensed. | gated |
| 6 | The trial is **REST-startable** with no UI: `POST {base}/login` (HTTP basic `admin:${ADMIN_PASSWORD}`, `Accept: application/json;api-version=1.0`) yields a bearer token; `POST {base}/license?action=start-trial-timer` with that token starts the listener. The image's `/Chariot/configure.sh` also honours `LICENSE_TYPE=online|floating` for a real license (there is no trial env). | automatable |
| 7 | **Closed-listener red herring:** before the license was started, the broker's MQTT port appeared "closed" — which read like a port/wiring bug. It was the license gate (finding 5), not the port. The in-network MQTT port is **1883** (matches `wires.mqtt.port`); `CHARIOT_MQTT_PORT=1886` in the manifest is only the host mapping. | resolved |
| 8 | **admin/changeme discovery:** the Chariot image auto-seeds an MQTT user `admin` with password `changeme` and full ACL — **not** tied to `ADMIN_PASSWORD` (which only sets the web admin). The seeds' JWE decrypts to `changeme`, which is why the captured blob works on any Chariot. An `MQTT_USERS` env (JSON → `POST /mqttusers`) exists for custom users. | discovered |
| 9 | **Trial caveats (demo posture):** Ignition's ~2h platform trial gates the Cirrus modules and resets on a gateway/container restart; Chariot's trial must be (re)started via the REST call (or UI) — the generated stack's `chariot-trial` init does this on every `up`. | documented |
| 10 | **Browser-not-supported:** the Ignition 8.3 gateway web UI rejects the headless Playwright browser (“Browser Not Supported”), so all evidence below was gathered via container logs, `docker`, and a `mosquitto_sub` broker capture rather than UI screenshots. | tooling note |

## Per-broker credential decisions

The four `mqtt-broker` catalog services all wire the same Cirrus modules, so they
share the seed trees, differing only in MQTT auth posture:

| Broker | Auth posture | Seed treatment | Live-verified? |
| --- | --- | --- | --- |
| `chariot` | image user `admin` / `changeme`, full ACL | full seed with the captured JWE password (decrypts to `changeme`) | **yes** |
| `emqx` | anonymous by default | same seed, `username`/`password` fields stripped | no (config-shaped) |
| `hivemq` (CE) | anonymous | same seed, credential fields stripped | no (config-shaped) |
| `rabbitmq` | MQTT plugin needs a user; its password cannot be JWE-encrypted at generation time | falls back to the credential-stripped seed + a POST-SETUP note | no (config-shaped) |

Only Chariot's seeded connection was proven live. The others are correct in
shape but unverified; their `POST-SETUP.md` step stays a manual procedure, while
Chariot's becomes a verification.

## Repro steps

The image was already cached; first boot to RUNNING was ~2 min/gateway.

```sh
# 1. Generate a hub-and-spoke + IIoT (chariot) stack with ignition-stack and
#    fetch the Cirrus modules into modules/cache/ (the README's generated/ used
#    project name "iiotspike").
ignition-stack init iiotspike --profile hub-and-spoke --iiot
ignition-stack modules download mqtt-engine mqtt-transmission

# 2. Capture: configure Transmission (spoke) + Engine (hub) once in the UI, then
#    diff data/config/resources/ before vs after to find what the gateway wrote
#    (captured/before vs captured/after; the *-{before,after}.txt are the find
#    listings). The distilled result is seed/.
#    NB: the 8.3 gateway UI rejects headless Playwright ("Browser Not
#    Supported"), so drive a real browser for the capture.

# 3. Replay: wipe volumes, drop seed/spoke and seed/hub into the gateways'
#    config/resources/ trees, start Chariot's trial, and bring it up.
docker compose down -v
# (seed files copied into services/<gw>/config/resources/…)
docker compose up -d

# 4. Verify without the UI: watch the logs and capture the broker.
docker logs iiotspike-spoke-1 | grep -E 'Connected to the MQTT Server|Publishing NBIRTH'
docker run --rm --network <stack>_default eclipse-mosquitto \
  mosquitto_sub -h chariot -p 1883 -u admin -P changeme -t '#' -v

# teardown (always -v to drop the volumes)
docker compose down -v
```
